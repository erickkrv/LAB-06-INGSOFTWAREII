Gestión de Configuración

La gestión de configuración es la disciplina que se encarga de identificar, controlar y dar seguimiento a todos los elementos que componen un sistema de software a lo largo de su ciclo de vida. En nuestro sistema de inventarios y ventas, esto abarca no solo el código fuente de la API (routers, services, models), sino también los esquemas de base de datos, las variables de entorno (settings.py, .env), las dependencias del proyecto (requirements.txt), la configuración del pipeline CI/CD (ci.yml), y la configuración de análisis estático (sonar-project.properties).

En un sistema que ya está operando en producción —como asumimos en este laboratorio—, la gestión de configuración es crítica porque un cambio no rastreado en cualquiera de estos elementos puede causar fallos en cascada. Por ejemplo, si alguien modifica settings.py para cambiar la DATABASE_URL sin documentarlo, el despliegue siguiente podría apuntar a una base de datos incorrecta y corromper datos reales de clientes y ventas.

Elementos que controlamos:

  • Código fuente: todos los módulos Python bajo models/, services/, routers/, schemas/, integrations/, utils/
  • Versiones del sistema: archivo VERSION y CHANGELOG.md
  • Estructura de datos: modelos SQLAlchemy y la base de datos inventario_ventas.db
  • Entornos y configuración: settings.py, .env.example, variables de entorno
  • Pipeline: .github/workflows/ci.yml
  • Dependencias: requirements.txt

Versionamiento del Software

El versionamiento semántico (SemVer) utiliza el formato MAJOR.MINOR.PATCH para comunicar la naturaleza de los cambios en cada release. Aplicado a nuestro sistema:

  • MAJOR (ej. 1.0.0 → 2.0.0): cambio que rompe la compatibilidad hacia atrás. Si dividimos el campo nombre de usuarios en nombre y apellido, los consumidores de la API reciben una respuesta con estructura distinta → rompe integraciones existentes.
  • MINOR (ej. 1.0.0 → 1.1.0): funcionalidad nueva sin romper lo existente. Agregar un endpoint de reportes de ventas por fecha no afecta a quienes ya usan checkout o productos.
  • PATCH (ej. 1.0.0 → 1.0.1): corrección de bugs sin funcionalidad nueva. Arreglar la vulnerabilidad de SQL injection en buscar_venta_por_cliente().

Romper compatibilidad significa que el contrato implícito entre la API y sus consumidores (frontends, integraciones externas) deja de ser válido. Esto obliga a los clientes a modificar su código para adaptarse, lo cual tiene un costo alto en producción.

Gestión de Cambios

Un cambio en producción es toda modificación deliberada al sistema que está siendo utilizado por usuarios reales. A diferencia de un cambio en desarrollo, tiene impacto directo e inmediato. Los clasificamos en tres tipos:

  • Cambio funcional: altera el comportamiento visible del sistema. Ejemplo: agregar paginación al listado de productos o permitir filtrar ventas por rango de fechas.
  • Cambio técnico: mejora interna que no cambia la funcionalidad visible. Ejemplo: reemplazar hashlib.md5 por bcrypt para el hashing de contraseñas en auth_service.py.
  • Cambio de datos: modifica la estructura o contenido de la base de datos. Ejemplo: agregar la tabla categorias y vincularla con productos, o dividir el campo nombre de usuarios.

Rollback

Un rollback es la acción de revertir el sistema a un estado anterior estable, deshaciendo los cambios de un despliegue problemático. Se utiliza cuando, después de un deploy, se detectan errores críticos que afectan la operación normal: por ejemplo, si tras migrar usuarios, el endpoint de checkout deja de funcionar porque no encuentra el campo nombre esperado.

Riesgos asociados:

  • Pérdida de datos: si se crearon registros nuevos con el esquema nuevo, revertir al viejo puede hacer que esos datos queden inaccesibles.
  • Inconsistencia: si la migración ya transformó datos (ej. dividió "Juan Pérez" en nombre="Juan", apellido="Pérez"), revertir el esquema no reconstruye el original.
  • Downtime extendido: el rollback puede tomar más tiempo que el despliegue original.

Migración de Datos (ETL / ELT)

ETL (Extract-Transform-Load): extrae datos del origen, los transforma en un sistema intermedio (limpia, mapea, valida), y luego los carga en el destino. Es más seguro porque la transformación ocurre fuera de producción. En nuestro sistema, extraeríamos los usuarios, transformaríamos el campo nombre dividiéndolo en Python, y cargaríamos los resultados en la nueva estructura.

ELT (Extract-Load-Transform): carga los datos al destino primero y transforma directamente en la BD destino usando SQL. Es más rápido pero más riesgoso porque opera sobre datos vivos.

Problemas comunes aplicables a nuestro sistema:

  • Duplicados: si la migración se ejecuta dos veces, podríamos insertar categorías o proveedores duplicados.
  • Inconsistencias de formato: el campo nombre puede tener "Juan Pérez", "MARIA GARCIA", " josé " — formatos variados que hacen que la división produzca resultados inesperados.
  • Datos incompletos: usuarios con nombre vacío o NULL causan errores en la transformación.
  • Relaciones inválidas: ventas cuyo usuario_id apunta a un usuario eliminado, o inventarios sin producto correspondiente.

b) Estrategia de Migración

Seleccionamos una migración gradual (por fases):

  • Fase 1 (v1.1.0): Cambios aditivos sin riesgo — campos opcionales con defaults, índices. Se despliegan primero como prerrequisito.
  • Fase 2 (v1.2.0): Cambios estructurales mayores — nuevas tablas, FKs, transformación de datos. Se ejecutan después de validar la Fase 1.
¿Por qué no Big Bang? Si todos los cambios se aplican de golpe y la división de nombres falla, quedaríamos con tablas a medio crear y datos parcialmente transformados. La migración gradual permite abortar entre fases sin haber tocado los datos más sensibles.

Esta decisión está alineada con la arquitectura de monolito modular en capas: cada fase corresponde a un conjunto coherente de cambios que puede validarse independientemente.

Enfoque híbrido ETL/ELT:

Datos existentes vs. nuevos: Los existentes se transforman durante la migración. Los nuevos se insertan directamente con la nueva estructura.

g) Idempotencia y Reprocesamiento

¿Se puede ejecutar la migración más de una vez sin efectos secundarios?

Sí. Los scripts implementan idempotencia mediante:

  • check_already_applied() consulta migration_history antes de ejecutar. Si ya está registrada, muestra [SKIP].
  • CREATE TABLE IF NOT EXISTS evita errores con tablas existentes.
  • ALTER TABLE ADD COLUMN con try/except que detecta "duplicate column".
  • INSERT OR IGNORE con constraints UNIQUE para datos semilla.

¿Cómo manejaríamos un fallo parcial?

  1. Identificar el paso fallido usando los mensajes de log (✔ o ❌)
  2. Corregir la causa (dato corrupto, constraint violado)
  3. Re-ejecutar la migración — los pasos ya completados se omiten por idempotencia

¿Es posible reprocesar sin duplicar?

Sí — la transformación usa UPDATE (no INSERT), las tablas usan IF NOT EXISTS, y los datos semilla usan INSERT OR IGNORE.

Estrategia de Rollback

a) Rollback de Aplicación (Código)

Para regresar a la versión anterior del sistema:

# Opción 1: Revertir al tag de la versión anterior
git checkout tags/v1.0.0

# Opción 2: Revert del merge commit
git revert <commit-hash-del-merge-de-release>

# Redeploy en ambos casos:
pip install -r requirements.txt
# Reiniciar uvicorn

El mecanismo principal son Git tags: cada release tiene su tag (v1.0.0, v1.1.0, v1.2.0), lo que permite revertir en segundos.

Precaución: Revertir el código sin revertir la BD causa incompatibilidad. El código v1.0.0 espera un campo nombre completo, pero tras la migración v1.2.0, ese campo solo contiene el primer nombre.

b) Rollback de Base de Datos (Estructura)

Cada script de migración incluye una función rollback():

# Revertir v1.2.0
python migrations/v1_2_0_breaking_changes.py --rollback

# Revertir v1.1.0
python migrations/v1_1_0_add_features.py --rollback
  • v1.1.0 (fácil): Reconstrucción de tablas sin columnas agregadas usando el patrón backup → drop → recrear → restaurar en SQLite.
  • v1.2.0 (parcial): Se eliminan tablas nuevas y se restauran usuarios desde la tabla de backup _backup_usuarios_v1_2_0.

c) Rollback de Datos (El Más Crítico)

Cambio¿Rollback posible?DificultadEstrategia
Agregar fecha_creacion Sí ✓ Fácil Se elimina la columna; datos eran valores por defecto
Agregar stock_minimo Sí ✓ Fácil Se elimina; valores eran defaults

Para cambios irreversibles, nuestra estrategia es:
  1. Corrección hacia adelante (forward-fix): nueva migración v1.2.1 que corrige el problema sin revertir.
  2. Hotfix: parche que acepta ambos formatos (nombre completo y nombre/apellido) temporalmente.
  3. Restaurar desde backup: último recurso — se pierden transacciones post-backup.

Escenario Crítico

Situación: 30 minutos después de ejecutar el despliegue v1.2.0, los vendedores reportan que el flujo de checkout falla con error 500. El sistema muestra "Internal Server Error" al registrar ventas.

a) Diagnóstico Inicial

Evidencia utilizada:

  1. Logs de uvicorn: 
    # En los logs encontramos:
sqlalchemy.exc.OperationalError: no such column: usuarios.nombre
File "services/checkout_service.py", line 23, in orquestar_checkout_completo
  user_db = db.query(Usuario).filter(Usuario.id == usuario_id).first()
  2. Métricas: Tasa de errores 500 subió de 0% a ~40% en los últimos 30 minutos.
  3. Comportamiento: Endpoints de productos e inventario SÍ funcionan. Solo los que involucran usuarios fallan → problema focalizado.

b) Identificación del Origen

Causa: Incompatibilidad código-datos. La migración v1.2.0 dividió nombre y agregó apellido, pero el modelo SQLAlchemy en models/usuario.py no incluyó el campo apellido. El ORM no puede mapear las columnas correctamente.

Tipo de problema: Código + Datos — La migración de datos fue correcta pero el código desplegado no corresponde al esquema actualizado.

c) Decisión Bajo Presión

Acción: Investigar primero (máximo 10 min), luego decidir.

Justificación:
  • No rollback inmediato: ya hay 30 min de transacciones con nueva estructura, se perderían.
  • No dejar así: vendedores no pueden operar → impacto directo al negocio.
  • Investigar rápido para determinar si es un fix simple o algo más profundo.

Acciones inmediatas:

  1. Comunicar: "Investigando error en checkout, ETA 10 minutos"
  2. Vendedores registran ventas manualmente como contingencia
  3. Revisar si usuario.py en el tag v1.2.0 tiene el campo apellido

d) Evaluación del Rollback

¿El rollback de 5.5 es aplicable? → Parcialmente, pero NO es la mejor opción.

OpciónViabilidadConsecuencia
Rollback completo (BD + código a v1.0.0) Posible pero costoso Se pierden 30 min de transacciones. Requiere restaurar backup.
Forward-fix (recomendado) Viable ✓ Agregar apellido = Column(String, default="") al modelo, commit, tag v1.2.1, redesplegar. ~5 minutos.
Rollback parcial (solo código a v1.1.0) No viable Código v1.1.0 no conoce las tablas nuevas ni el esquema modificado.

Decisión final: Forward-fix. Se actualiza el modelo, se tagea v1.2.1, y se redesplega. Checkout funciona sin perder datos.

PreguntaRespuesta

Validación post-despliegue

  • Health check: GET /
  • Flujo de checkout completo de prueba
  • Verificar usuarios con nombre/apellido
  • Revisión de logs de aplicación

Criterio de abortar

  • Migración retorna [ERROR]
  • Validación tiene resultado FAIL
  • Checkout falla (error 500)
  • >5% de usuarios con errores de transformación
  • API no responde en 30 segundos post-restart

Punto de no retorno

Una vez que se crean transacciones nuevas con la estructura v1.2.0, revertir significaría perder esos datos. La estrategia cambia de "revertir" a "corregir hacia adelante".

Reflexión Final

¿Qué problemas no detecta el pipeline CI/CD?

  • Incompatibilidad modelos ↔ esquema de BD: Los tests usan BD en memoria creada desde los modelos, siempre coinciden. En producción, la BD real puede tener esquema diferente tras migraciones manuales.
  • Problemas de rendimiento bajo carga: Una consulta que funciona en 10ms con 100 registros puede tardar 30 segundos con 100,000 en producción.
  • Fallos en integraciones externas: Los mocks siempre retornan éxito. En producción, la pasarela de pagos puede fallar por timeout o cambios en su API.
  • Errores de datos reales: Los tests usan datos perfectos y controlados. Los datos reales tienen formatos inesperados, caracteres especiales, valores extremos.

¿Qué problemas solo aparecen en producción?

  • Condiciones de carrera: Dos vendedores comprando el último producto en stock simultáneamente → ambos ven stock=1 → ambos descuentan → stock=-1.
  • Datos acumulados: La BD crece; queries lentos sin índices adecuados.
  • Configuración de entorno: Variables de entorno mal configuradas, diferencias entre SO del CI (Ubuntu) y servidor de producción.
  • Fallos intermitentes: La pasarela simulada falla 5% del tiempo, pero en producción real podría ser 15% en horarios pico.

¿Qué fue más difícil: diseñar el sistema o migrarlo?

Migrarlo fue significativamente más difícil. Diseñar desde cero permite tomar decisiones libremente, sin restricciones ni datos existentes. Migrar en producción implica:

  • No perder ni un solo registro de ventas o facturas (documentos legales/contables)
  • Transformar datos con formatos inconsistentes acumulados durante meses de operación
  • Mantener compatibilidad temporal durante la transición
  • Coordinar código, estructura de BD y datos en una ventana de tiempo limitada
  • Tener un plan B (rollback) para cada posible falla

El diseño es como construir una casa nueva; la migración es como remodelarla mientras la familia sigue viviendo adentro.

¿Qué decisión consideran más riesgosa?

La división del campo nombre de usuarios es la más riesgosa por tres razones:

  1. Transformación destructiva: "Juan Carlos de la Fuente Rodríguez" se divide en nombre="Juan" y apellido="Carlos de la Fuente Rodríguez", pero la intención original podría haber sido otra. No hay algoritmo que lo resuelva con certeza.
  2. Impacto transversal: todos los componentes que muestran, buscan o procesan nombres deben actualizarse simultáneamente.
  3. Rollback parcial: usuarios creados post-migración ya tienen nombre/apellido separados; revertir requiere concatenar y se pierde la estructura.
