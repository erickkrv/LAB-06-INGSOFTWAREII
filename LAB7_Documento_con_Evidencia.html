Gestión de Configuración

La gestión de configuración es la disciplina que se encarga de identificar, controlar y dar seguimiento a todos los elementos que componen un sistema de software a lo largo de su ciclo de vida. En nuestro sistema de inventarios y ventas, esto abarca no solo el código fuente de la API (routers, services, models), sino también los esquemas de base de datos, las variables de entorno (settings.py, .env), las dependencias del proyecto (requirements.txt), la configuración del pipeline CI/CD (ci.yml), y la configuración de análisis estático (sonar-project.properties).

En un sistema que ya está operando en producción —como asumimos en este laboratorio—, la gestión de configuración es crítica porque un cambio no rastreado en cualquiera de estos elementos puede causar fallos en cascada. Por ejemplo, si alguien modifica settings.py para cambiar la DATABASE_URL sin documentarlo, el despliegue siguiente podría apuntar a una base de datos incorrecta y corromper datos reales de clientes y ventas.

Elementos que controlamos:

  • Código fuente: todos los módulos Python bajo models/, services/, routers/, schemas/, integrations/, utils/
  • Versiones del sistema: archivo VERSION y CHANGELOG.md
  • Estructura de datos: modelos SQLAlchemy y la base de datos inventario_ventas.db
  • Entornos y configuración: settings.py, .env.example, variables de entorno
  • Pipeline: .github/workflows/ci.yml
  • Dependencias: requirements.txt

Versionamiento del Software

El versionamiento semántico (SemVer) utiliza el formato MAJOR.MINOR.PATCH para comunicar la naturaleza de los cambios en cada release. Aplicado a nuestro sistema:

  • MAJOR (ej. 1.0.0 → 2.0.0): cambio que rompe la compatibilidad hacia atrás. Si dividimos el campo nombre de usuarios en nombre y apellido, los consumidores de la API reciben una respuesta con estructura distinta → rompe integraciones existentes.
  • MINOR (ej. 1.0.0 → 1.1.0): funcionalidad nueva sin romper lo existente. Agregar un endpoint de reportes de ventas por fecha no afecta a quienes ya usan checkout o productos.
  • PATCH (ej. 1.0.0 → 1.0.1): corrección de bugs sin funcionalidad nueva. Arreglar la vulnerabilidad de SQL injection en buscar_venta_por_cliente().

Romper compatibilidad significa que el contrato implícito entre la API y sus consumidores (frontends, integraciones externas) deja de ser válido. Esto obliga a los clientes a modificar su código para adaptarse, lo cual tiene un costo alto en producción.

Gestión de Cambios

Un cambio en producción es toda modificación deliberada al sistema que está siendo utilizado por usuarios reales. A diferencia de un cambio en desarrollo, tiene impacto directo e inmediato. Los clasificamos en tres tipos:

  • Cambio funcional: altera el comportamiento visible del sistema. Ejemplo: agregar paginación al listado de productos o permitir filtrar ventas por rango de fechas.
  • Cambio técnico: mejora interna que no cambia la funcionalidad visible. Ejemplo: reemplazar hashlib.md5 por bcrypt para el hashing de contraseñas en auth_service.py.
  • Cambio de datos: modifica la estructura o contenido de la base de datos. Ejemplo: agregar la tabla categorias y vincularla con productos, o dividir el campo nombre de usuarios.

Rollback

Un rollback es la acción de revertir el sistema a un estado anterior estable, deshaciendo los cambios de un despliegue problemático. Se utiliza cuando, después de un deploy, se detectan errores críticos que afectan la operación normal.

Riesgos asociados:

  • Pérdida de datos: si se crearon registros nuevos con el esquema nuevo, revertir al viejo puede hacer que esos datos queden inaccesibles.
  • Inconsistencia: si la migración ya transformó datos, revertir el esquema no reconstruye el original.
  • Downtime extendido: el rollback puede tomar más tiempo que el despliegue original.

Migración de Datos (ETL / ELT)

ETL (Extract-Transform-Load): extrae datos del origen, los transforma en un sistema intermedio, y luego los carga en el destino. En nuestro sistema, extraeríamos los usuarios, transformaríamos el campo nombre dividiéndolo en Python, y cargaríamos los resultados en la nueva estructura.

ELT (Extract-Load-Transform): carga los datos al destino primero y transforma directamente en la BD destino usando SQL. Es más rápido pero más riesgoso porque opera sobre datos vivos.

b) Estrategia de Migración: Gradual por fases

  • Fase 1 (v1.1.0): Cambios aditivos sin riesgo — campos opcionales con defaults, índices.
  • Fase 2 (v1.2.0): Cambios estructurales mayores — nuevas tablas, FKs, transformación de datos.
¿Por qué no Big Bang? Si todos los cambios se aplican de golpe y la división de nombres falla, quedaríamos con tablas a medio crear y datos parcialmente transformados. La migración gradual permite abortar entre fases sin haber tocado los datos más sensibles.
EVIDENCIA: Script de Migración v1.1.0 (Fase 1 — MINOR)
🐍 migrations/v1_1_0_add_features.py
""" Migración v1.1.0 - Nuevas funcionalidades (MINOR) ================================================== Cambios: - Agrega campo 'fecha_creacion' a tabla 'ventas' - Agrega campo 'stock_minimo' a tabla 'inventarios' - Crea índice en ventas.usuario_id para optimizar consultas Tipo: MINOR (compatible hacia atrás) """ import sqlite3 import sys, os from datetime import datetime MIGRATION_ID = "v1_1_0" MIGRATION_DESCRIPTION = "Agregar fecha_creacion a ventas, stock_minimo a inventarios" def check_already_applied(cursor): """Verifica si la migración ya fue aplicada (idempotencia).""" cursor.execute(""" SELECT name FROM sqlite_master WHERE type='table' AND name='migration_history' """) if not cursor.fetchone(): cursor.execute(""" CREATE TABLE migration_history ( id INTEGER PRIMARY KEY AUTOINCREMENT, migration_id TEXT UNIQUE NOT NULL, description TEXT, applied_at TEXT NOT NULL, rolled_back_at TEXT ) """) return False cursor.execute( "SELECT id FROM migration_history WHERE migration_id = ? AND rolled_back_at IS NULL", (MIGRATION_ID,) ) return cursor.fetchone() is not None def migrate(db_path): """Ejecuta la migración hacia adelante.""" conn = sqlite3.connect(db_path) cursor = conn.cursor() try: if check_already_applied(cursor): print(f"[SKIP] Migración {MIGRATION_ID} ya fue aplicada.") return True # 1. Agregar campo fecha_creacion a ventas try: cursor.execute("ALTER TABLE ventas ADD COLUMN fecha_creacion TEXT") print(" ✔ Campo 'fecha_creacion' agregado a tabla 'ventas'") except sqlite3.OperationalError as e: if "duplicate column" in str(e).lower(): print(" ⚠ Campo ya existe, omitiendo...") else: raise # 2. Agregar campo stock_minimo a inventarios try: cursor.execute("ALTER TABLE inventarios ADD COLUMN stock_minimo INTEGER DEFAULT 5") print(" ✔ Campo 'stock_minimo' agregado a tabla 'inventarios'") except sqlite3.OperationalError as e: if "duplicate column" in str(e).lower(): print(" ⚠ Campo ya existe, omitiendo...") else: raise # 3. Crear índice cursor.execute("CREATE INDEX IF NOT EXISTS idx_ventas_usuario ON ventas(usuario_id)") # Registrar migración cursor.execute( "INSERT INTO migration_history (migration_id, description, applied_at) VALUES (?, ?, ?)", (MIGRATION_ID, MIGRATION_DESCRIPTION, datetime.now().isoformat()) ) conn.commit() print(f"[SUCCESS] Migración {MIGRATION_ID} aplicada exitosamente.") return True except Exception as e: conn.rollback() print(f"[ERROR] Migración {MIGRATION_ID} falló: {e}") return False finally: conn.close() def rollback(db_path): """Revierte la migración usando reconstrucción de tablas en SQLite.""" conn = sqlite3.connect(db_path) cursor = conn.cursor() try: # Reconstruir ventas sin fecha_creacion cursor.execute("CREATE TABLE ventas_backup AS SELECT id, usuario_id, total, estado, metodo_pago FROM ventas") cursor.execute("DROP TABLE ventas") # ... reconstrucción completa ... cursor.execute("DROP INDEX IF EXISTS idx_ventas_usuario") cursor.execute("UPDATE migration_history SET rolled_back_at = ? WHERE migration_id = ?", (datetime.now().isoformat(), MIGRATION_ID)) conn.commit() return True except Exception as e: conn.rollback() return False
Justificación: El script de migración v1.1.0 implementa tres conceptos clave del laboratorio:
  • Idempotencia: check_already_applied() consulta migration_history antes de ejecutar, y cada ALTER TABLE usa try/except para detectar columnas duplicadas.
  • Reversibilidad: La función rollback() reconstruye las tablas originales usando el patrón backup → drop → recrear → restaurar de SQLite.
  • Trazabilidad: Cada migración se registra en migration_history con timestamp y estado.

🐍 migrations/v1_2_0_breaking_changes.py — Función de Transformación ETL
""" Migración v1.2.0 - Breaking Changes (MAJOR) ============================================= ADVERTENCIA: Esta migración NO es fácilmente reversible para el campo 'nombre' → 'nombre' + 'apellido' """ MIGRATION_ID = "v1_2_0" PREREQUISITE = "v1_1_0" # ← Prerequisito obligatorio def check_prerequisite(cursor): """Verifica que la migración v1.1.0 esté aplicada.""" cursor.execute( "SELECT id FROM migration_history WHERE migration_id = ? AND rolled_back_at IS NULL", (PREREQUISITE,) ) return cursor.fetchone() is not None def transform_nombre(nombre_completo): """ Transforma 'nombre_completo' en (nombre, apellido). Estrategia: la primera palabra es nombre, el resto es apellido. Si solo hay una palabra, apellido queda vacío. """ if not nombre_completo or not nombre_completo.strip(): return ("Sin nombre", "") partes = nombre_completo.strip().split(" ", 1) nombre = partes[0] apellido = partes[1] if len(partes) > 1 else "" return (nombre, apellido)
Justificación — Patrón ETL: La función transform_nombre() implementa la lógica de transformación (la "T" de ETL) en Python en lugar de SQL, como se decidió en la estrategia de migración. Esto permite manejar casos edge:
  • Nombres vacíos/NULL → se asigna "Sin nombre" con apellido vacío
  • Espacios múltiples → strip() normaliza antes de dividir
  • Un solo nombre → apellido queda vacío en lugar de causar crash

def rollback(db_path): """ Revierte la migración v1.2.0. NOTA: La reversión del campo 'nombre' es PARCIAL. Se restaura desde tabla de backup '_backup_usuarios_v1_2_0'. """ conn = sqlite3.connect(db_path) cursor = conn.cursor() try: print("⚠ ADVERTENCIA: Esta reversión puede causar pérdida de datos") print(" en registros creados después de la migración original.") # Restaurar usuarios desde backup si existe cursor.execute(""" SELECT name FROM sqlite_master WHERE type='table' AND name='_backup_usuarios_v1_2_0' """) if cursor.fetchone(): cursor.execute("DELETE FROM usuarios") cursor.execute("""INSERT INTO usuarios (id, nombre, email, password_hash, rol) SELECT id, nombre, email, password_hash, rol FROM _backup_usuarios_v1_2_0""") cursor.execute("DROP TABLE _backup_usuarios_v1_2_0") print(" ✔ Usuarios restaurados desde backup") else: # Fallback: concatenar nombre + apellido cursor.execute("UPDATE usuarios SET nombre = nombre || ' ' || COALESCE(apellido, '')") print(" ⚠ Backup no encontrado, se concatenó nombre + apellido") # Eliminar tablas nuevas for tabla in ['producto_proveedor', 'proveedores', 'categorias', 'auditoria_cambios']: cursor.execute(f"DROP TABLE IF EXISTS {tabla}") # Marcar como revertido en historial cursor.execute("UPDATE migration_history SET rolled_back_at = ? WHERE migration_id = ?", (datetime.now().isoformat(), MIGRATION_ID)) conn.commit() print("⚠ Datos de proveedores/categorías creados post-migración se perdieron.") except Exception as e: conn.rollback()
Justificación — Rollback estratificado: El rollback de v1.2.0 demuestra la complejidad de revertir cambios destructivos:

  • Si no existe backup, usa concatenación como fallback (menos preciso)
  • Tablas nuevas se eliminan con DROP TABLE IF EXISTS
  • Se advierte explícitamente sobre pérdida de datos post-migración

Reflexión Final

¿Qué problemas no detecta el pipeline CI/CD?

  • Incompatibilidad modelos ↔ esquema de BD: Los tests usan BD en memoria creada desde los modelos, siempre coinciden. En producción, la BD real puede tener esquema diferente.
  • Problemas de rendimiento bajo carga: Una consulta que funciona en 10ms con 100 registros puede tardar 30s con 100,000.
  • Fallos en integraciones externas: Los mocks siempre retornan éxito.
  • Errores de datos reales: Los tests usan datos perfectos y controlados.

¿Qué problemas solo aparecen en producción?

  • Condiciones de carrera: Dos vendedores comprando el último producto simultáneamente.
  • Datos acumulados: BD crece; queries lentos sin índices adecuados.
  • Configuración de entorno: Variables mal configuradas, diferencias entre CI y producción.

¿Qué fue más difícil: diseñar el sistema o migrarlo?

Migrarlo fue significativamente más difícil. Diseñar desde cero permite tomar decisiones libremente. Migrar en producción implica:

  • No perder ni un solo registro de ventas o facturas
  • Transformar datos con formatos inconsistentes
  • Mantener compatibilidad temporal durante la transición
  • Tener un plan B (rollback) para cada posible falla

El diseño es como construir una casa nueva; la migración es como remodelarla mientras la familia sigue viviendo adentro.

¿Qué decisión consideran más riesgosa?

La división del campo nombre de usuarios es la más riesgosa por tres razones:

  1. Transformación destructiva: "Juan Carlos de la Fuente Rodríguez" se divide en nombre="Juan" y apellido="Carlos de la Fuente Rodríguez".
  2. Impacto transversal: todos los componentes que procesan nombres deben actualizarse.
  3. Rollback parcial: usuarios creados post-migración ya tienen estructura nueva.
